Design Philosophy

Two times of day, one library

The interface is a wizard's library. The dark theme is the same library after sunset, lit by candle. The light theme is morning, sunlight on parchment. Same world, different hour.

Candlelit, after dark

Quiet darkness with warm illumination. Cards glow on the table; ornate borders catch the firelight. The mood is contemplation, late-night study, a single tome open under candleflame. This is the default — it's where the project lives most of the time.

Morning, sun on parchment

Sunlit warmth on cream paper. The same illuminated borders softened by daylight; ink turned to sepia rather than near-black. The mood is unhurried mid-morning, sun through tall windows on hand-painted manuscripts. Optional — for readers who find the dark oppressive.

Andika
body text · UI labels · accessibility-first

Andika was designed with literacy and dyslexia in mind. Its letterforms are unambiguous — the lowercase a is a single-storey form, the I, l, and 1 are visually distinct, and the round shapes are open. It sets quietly at a long reading length without fatigue.

In this product it carries everything functional: card text, oracle rules, filter labels, search results, settings, helper copy. The visual character belongs to Cinzel and the ornament; Andika's job is to disappear into legibility.

Layout Principles

How the page behaves

Four rules that hold across every screen. They're what makes this design read as calm instead of cluttered, even when the data underneath is dense.

I.
Only one layer is loud at a time
The design has three aesthetic layers — atmospheric darkness, ornate parchment chrome, illuminated card content. Only one of them gets to dominate any given screen. If the chrome is ornate, cards stay restrained. If cards are the focus, chrome quiets to thin lines and soft borders.
II.
Filter-first, firehose-never
The interface always starts narrow. The user picks one filter, then another, then another — each one revealing a smaller set of cards. No infinite-scrolling lists, no overwhelming columns of options on first view. The opposite of Moxfield's everything-at-once.
III.
Generous breathing room
Padding is more generous than feels strictly necessary. Negative space is what gives the parchment its weight — without it, the page reads as a database printout. Sixty-pixel gaps between sections, twenty-four between cards, sixteen inside them.
IV.
Accessibility is structural, not patched
Semantic HTML, focus management on state changes, ARIA matched to W3C patterns, visible focus rings, prefers-reduced-motion respected, contrast above WCAG AA in both themes. Audited as the component is built, not later. Fake accessibility is worse than none.

When Each Piece Ships

Phase map for this design

The design lands across the project's phases — some of it as scaffolding in Phase 1, some of it as features that arrive when their phase does.

Phase 1A

Foundation & display chrome

Dark candlelit palette applied as the default; typography pair loaded; sample card display component; navigation; calm layout primitives. Enough visual identity to ship the MVP, but the theme architecture beneath stays scrappy.

Phase 1B

Theme architecture & polish

CSS variables formalised; theme tokens ready for the eventual swap; dark theme tuned for contrast, hierarchy, and accessibility — meets WCAG, not just "works."

Phase 2

Deck-builder visualisations

Mana curve, colour distribution, type breakdown — each shown calmly, one at a time behind tabs rather than all metrics on screen at once. Commander spotlight uses the same card display chrome at a larger scale.

Phase 3

AI surface

Conversational pane with the AI Orchestrator, deck-suggestion cards that match the same card display component, response chrome that fits the manuscript voice. BYOK settings page styled to match the rest of the system.

Phase 4

Morning library light palette

The light theme palette ships properly. The architecture has been in place since Phase 1B, so this becomes a content release, not a rebuild. Toggle lives in user settings or header.

Phase 5

Integration surfaces

OCR scanning UI, multi-provider AI settings, public-facing pages if the project goes open. All inherit the same design voice — no separate design system per feature.
